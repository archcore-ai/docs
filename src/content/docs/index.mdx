Archcore is an open-source **System Context Platform**. It structures your project's architectural knowledge into documents that AI coding agents can discover, read, and follow through MCP.

## The Problem

AI coding agents (Claude Code, Cursor, Copilot, etc.) produce better results when they have context about your project — decisions, rules, conventions, patterns. Today, teams store this context in:

- `CLAUDE.md` / `.cursorrules` — flat files that grow unwieldy
- Confluence / Notion — not accessible to agents
- People's heads — not accessible to anyone else

None of these scale. As context grows, flat files become hard to maintain. As teams grow, knowledge gets fragmented.

## The Solution

Archcore provides:

1. **10 document types** — each with a specific purpose, required sections, and a template
2. **Free-form directory structure** — organize by domain, feature, team, or however makes sense
3. **Virtual categories** — documents are classified by their type (in the filename), not by where they live
4. **Relations** — explicit links between documents (implements, extends, depends_on, related)
5. **MCP server** — agents read and write documents through 8 MCP tools
6. **Validation** — CLI checks structure, naming, frontmatter, and relations
7. **Multi-agent support** — one setup works with Claude Code, Cursor, Copilot, Gemini CLI, and more

## Three Categories

Every document type belongs to one of three categories:

### Knowledge
Decisions, standards, and reference material. The foundation your team builds on.

| Type | Purpose |
|------|---------|
| **adr** | Architecture Decision Record — records a decision that was made |
| **rfc** | Request for Comments — proposes a change for review |
| **rule** | Mandatory standard or required behavior |
| **guide** | Step-by-step instructions for a task |
| **doc** | General reference documentation |

### Vision
Where the product and project are heading.

| Type | Purpose |
|------|---------|
| **prd** | Product Requirements Document |
| **idea** | Product or technical concept to explore |
| **plan** | Implementation plan with phased tasks |

### Experience
Patterns learned from doing the work.

| Type | Purpose |
|------|---------|
| **task-type** | Proven workflow for a recurring task |
| **cpat** | Code pattern change — documents how and why a convention changed |

## How It Works

```
Your project
├── .archcore/
│   ├── settings.json
│   ├── auth/
│   │   ├── jwt-strategy.adr.md        ← knowledge
│   │   └── auth-redesign.prd.md       ← vision
│   ├── payments/
│   │   └── stripe-integration.guide.md ← knowledge
│   └── onboarding-flow.task-type.md   ← experience
└── src/
    └── ...
```

When an agent starts a session, Archcore's MCP server makes all documents available through tools like `list_documents`, `get_document`, and `create_document`. The agent can query, read, and even create new documents without leaving the conversation.
